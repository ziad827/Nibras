CLI Setup Guide

Install the Nibras CLI, authenticate with GitHub, and submit your first project — step by step.

Prerequisites

Node.js ≥ 18, npm ≥ 9, and git

You need Node.js ≥ 18, npm ≥ 9, and git before installing the CLI. Follow the steps for your OS.

① Install Node.js ≥ 18 + npm

# Recommended — nvm (no sudo, no permission issues)
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.39.7/install.sh | bash
source ~/.zshrc          # zsh default on macOS 10.15+ — use ~/.bashrc for bash
nvm install --lts
nvm use --lts
nvm alias default node

# Alternative — Homebrew
# brew install node
nvm installs Node.js entirely in your home directory — no sudo or permission fixes ever needed. If you already have a system Node.js and hit EACCES, see step 10.

② Install Git

# Xcode Command Line Tools (built-in to macOS)
xcode-select --install

# Or via Homebrew:
# brew install git

③ Verify all three tools

node --version   # must be v18.x or higher
npm --version    # must be 9.x or higher
git --version

All three commands must return version numbers. If any fails, revisit the install step above and open a new terminal after installing.

02

Install the CLI

Install the current CLI release globally

Install @nibras/cli globally via npm or the release install script. This makes the nibras command available everywhere in your terminal.

Local full-service repo: from the Nibras monorepo root run npm run build:cli:package && npm link to install the in-repo CLI against the gateway at http://localhost:8080.

Option A — install script (recommended)

curl -fsSL "https://github.com/EpitomeZied/nibras/releases/download/v2.0.0/install.sh" | bash

Option B — npm

npm install -g @nibras/cli@2.0.0

If the install fails with EEXIST or ENOTDIR, you have a stale global link. Remove it and reinstall:

npm uninstall -g nibras @nibras/cli || true
rm -f "$(npm config get prefix)/bin/nibras"
rm -rf "$(npm root -g)/nibras" "$(npm root -g)/@nibras/cli"
npm install -g @nibras/cli@2.0.0

03

Authenticate with GitHub

Log in with your GitHub account

Use an explicit --api-base-url for hosted onboarding so the CLI targets this deployment instead of the local dev default at http://127.0.0.1:4848. The login flow prints a one-time URL and short code, then tries to open the browser automatically unless you pass --no-open. After GitHub authorization, the CLI also prompts you to install the Nibras GitHub App so submissions and repo sync work.

On Windows, the most common path is: run the login command in Git Bash, let the browser open, approve GitHub, then return to the same terminal window and wait for the success box.

If the browser does not open, rerun with nibras login --api-base-url https://nibras-api.fly.dev --no-open and paste the printed URL into any browser manually.

nibras login --api-base-url https://nibras-api.fly.dev

Credentials are saved in ~/Library/Application Support/nibras/config.json (macOS), ~/.config/nibras/config.json (Linux), or %APPDATA%\nibras\config.json (Windows).

04

Create a Course

Set up your course and projects on the web dashboard

Before students can use the CLI, create a course and at least one project from the web dashboard.

  1. Go to Instructor → New Course
  2. Enter the course code (e.g. cs101), title, and term
  3. Add a project with an allowed file path pattern
  4. Copy the project key (e.g. cs101/assignment-1)

06

Run Tests Locally

Run the project test suite

Use nibras test to run the manifest-configured test command for your OS. Pass --previous only when the project manifest supports it.

nibras test
nibras test --previous   # include previous milestone tests

A non-zero exit means the configured test command failed. Projects that do not opt in to previous-milestone runs will reject --previous.

07

Submit Your Work

Stage, commit, push, and verify

nibras submit runs the configured local test command, stages only allowed files, creates a commit, pushes to origin, registers the submission, and polls for server-side verification.

nibras submit
  Running tests: npm test
  ✓ Staged 3 files
  ✓ Pushed commit a3f7c1d
  Verifying  ████████████████░░░░  80%

╭──────────────────────────────╮
│  ✓  Submission passed        │
│  Status:  passed             │
╰──────────────────────────────╯
Only files matching submission.allowedPaths in your manifest are included — students can't accidentally submit test or grading files.

A failing local test run does not abort submission. The CLI records the local result and server-side verification continues regardless.
08

Check Status

Diagnostic commands

Two diagnostic commands are available at any time:

nibras ping    # check API, auth, GitHub, GitHub App, and project status
nibras whoami  # show signed-in user and linked GitHub account

nibras ping is the fastest way to diagnose any problem — run it first. When run inside a project directory it also shows the project key and origin remote.

Troubleshooting

Common issues and fixes

Start with nibras ping — it checks API reachability, auth, GitHub linkage, and App install status in one command. Use the OS tab below to filter fixes for your platform.

"nibras: command not found" after install

The npm global bin directory is not in PATH, or the terminal was not restarted after install.

npm config get prefix
# Add the bin folder to PATH:
export PATH="$(npm config get prefix)/bin:$PATH"
# Make it permanent — append to ~/.zshrc

With nvm, the global bin is always in PATH automatically — no manual steps.

"EACCES: permission denied" on global install

System Node.js requires root to write to the global npm directory. Never use sudo npm install -g.

# Recommended: switch to nvm
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.39.7/install.sh | bash
source ~/.zshrc && nvm install --lts && nvm use --lts
npm install -g @nibras/cli@2.0.0

# Alternative: fix the npm global prefix without sudo
mkdir -p ~/.npm-global && npm config set prefix '~/.npm-global'
export PATH="$HOME/.npm-global/bin:$PATH"
"EEXIST" or "ENOTDIR" — stale global link

An older Nibras install left behind a broken symlink or directory.

npm uninstall -g nibras @nibras/cli || true
rm -f "$(npm config get prefix)/bin/nibras"
rm -rf "$(npm root -g)/nibras" "$(npm root -g)/@nibras/cli"
npm install -g @nibras/cli@2.0.0

Session token expired, revoked, or wrong API URL saved.

nibras login --api-base-url https://nibras-api.fly.dev

CLI Command Reference

Use this appendix for commands outside the main onboarding path or when you need the full CLI surface at a glance.

Core workflow

nibras login --api-base-url <api-url>
Start hosted device login against a specific Nibras deployment. Use the explicit API URL for hosted onboarding. The CLI default remains the local dev API.
nibras setup --project <key>
Bootstrap or refresh a local project.
nibras task
Print cached task text or fetch it if missing.
nibras test
Run the manifest test command.
nibras submit
Test, stage allowed files, commit, push, and wait for verification.

Discovery

nibras list
List all enrolled courses and their projects.
nibras status
Show recent submissions with live status badges.

Session and diagnostics

nibras whoami
Show the signed-in user and linked GitHub account.
nibras ping
Show API, auth, GitHub, GitHub App, and project status. Inside a project, it also reports the project key and origin remote.
nibras doctor
Run local environment and connectivity checks.
nibras config
Show or update CLI config (path, get, set).
nibras milestones
List milestones for the current project or a given project key.
nibras logout
Clear the local session.

Install lifecycle

nibras update --version <tag>
Reinstall a pinned published CLI release. Use nibras update --check to compare the installed CLI against the latest GitHub release.
nibras update --force --version <tag>
Force reinstall the same pinned tag.
nibras uninstall
Remove the global CLI install and keep local config.

Advanced / compatibility

nibras update-buildpack --node <version>
Edit the Node buildpack version in .nibras/project.json. Advanced manifest maintenance only; standard students usually do not need this.
nibras legacy ...
Run the older CLI entrypoint for compatibility. Compatibility-only. It is not part of the hosted default workflow.
